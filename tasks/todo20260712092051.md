# Issue #95 — did:webvh hash wire-format conformance

## Scope and acceptance criteria

- [x] Replace the non-conformant codec-plus-digest encoding used for did:webvh SCIDs,
      entry hashes, and `nextKeyHashes` commitments with a complete SHA-256 multihash
      (`0x12 0x20 <32-byte digest>`).
- [x] Encode that multihash as bare base58btc, without the multibase `z` prefix, at all
      three writer/validator call sites.
- [x] Keep the three formats on one shared encoding primitive so they cannot drift apart.
- [x] Pin independent known-answer values for SCID, entry hash, and key commitment, and
      assert the decoded wire shape (46 base58 characters / 34 multihash bytes where useful).
- [x] Preserve end-to-end create, update, resolve, and pre-rotation validation behavior under
      the new intentionally breaking wire format; old non-conformant logs should no longer be
      accepted as though they were v1.0 logs.
- [x] Correct the independent v1.0 genesis flow: compute the SCID with `versionId: "{SCID}"`,
      substitute the SCID, compute a distinct genesis entry hash using the SCID as predecessor,
      and publish `versionId: "1-<entryHash>"`; validate both hashes independently.
- [x] Correct subsequent-entry hashing so the hash input's `versionId` is exactly the previous
      entry's full published `versionId`, without prepending the new entry's version number.
- [x] Update the PRD and changelog to state that did:webvh v1.0 hashes are bare base58btc
      encodings of complete multihashes, document the corrected genesis flow, and record the
      breaking compatibility impact.
