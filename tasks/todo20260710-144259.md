## Review

**Status: complete. 784 tests pass (all 6 projects incl. W3C conformance); did:webvh sample runs clean.**

### Delivered (in this PR — issue #82 DID-binding/authority class)
- `DidUpdateResult.AuthorizationChanged` (bool, additive, default false); did:webvh sets it by comparing
  effective `updateKeys` / `nextKeyHashes` / `prerotation` / `witness` before vs. after the update (`ttl` excluded).
- `UpdateCoreAsync` / `DeactivateCoreAsync`: reject a deactivated log; reject a log whose latest `state.id != did`;
  reject a log whose **genesis SCID != the SCID in `did`**; (Update) reject `NewDocument.Id != did`.
- Resolve: bind the DID's self-certifying SCID to the genesis (`ExtractScid(did) == entries[0].Parameters.Scid`),
  closing a **critical** self-certification-bypass found by the adversarial audit (a host/CDN/MITM adversary could
  otherwise serve a self-signed genesis claiming the victim DID). Same root cause as the write-path binding.
- 14 Issue82 tests (binding, deactivated-log, auth-change true/false, and forged-genesis SCID repros on
  resolve/update/deactivate).

### Adversarial audit
Ran an 8-lens red-team with per-finding adversarial verification on the completed code (mandated by CLAUDE.md §2).
Two findings were the #82 class and are fixed here (resolve + write-path SCID binding). The audit also surfaced
additional CONFIRMED vulnerabilities in **other subsystems** (witness enforcement, HTTP client, serialization) that
are out of scope for this PR and were reported to the maintainer separately (kept out of public artifacts pending
fix, to avoid disclosing unpatched issues). See the session security memo.
