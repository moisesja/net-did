## Review

- Verified locally with `dotnet test tests/NetDid.Core.Tests/NetDid.Core.Tests.csproj --filter "DefaultBbsCryptoProviderTests|DefaultCryptoProviderTests|DefaultKeyGeneratorTests"` and `cargo test` in `native/zkryptium-ffi/`.
- Findings:
- `DefaultCryptoProvider` uses the `BLS_SIG_BLS12381G2_XMD:SHA-256_SSWU_RO_NUL_` DST for both BLS variants, so `KeyType.Bls12381G2` signs and verifies with the wrong ciphersuite domain.
- The BBS+ package/runtime story is not yet dependable across the stated surface: `NetDid.Core.csproj` packs whatever native assets exist, but the repo currently contains only `src/NetDid.Core/runtimes/osx-arm64/native/libzkryptium_ffi.dylib`.
- `DefaultBbsCryptoProvider` hard-codes a 4096-byte proof buffer and offers no retry/allocation path for larger valid proofs.
- The Rust FFI layer does not validate output pointers before writing through them, so the exported C ABI is not hardened against null or invalid outputs from non-.NET callers.
- Residual risk:
- Current tests prove same-library round trips on the current machine, but they do not prove interoperability between `DefaultKeyGenerator`'s Nethermind-generated BLS G2 keys and `DefaultBbsCryptoProvider`'s zkryptium-backed BBS+ operations, nor cross-implementation compatibility for the BLS G2 variant.
