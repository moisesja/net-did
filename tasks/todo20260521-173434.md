# Issues #48 + #51 — Conformance scope + did:webvh HTTP resource limits

## Branch
`fix/issues-48-and-51-resource-limits-and-conformance-scope`

## Issue #51 — HTTP resource limits

### Vulnerability summary

`DefaultWebVhHttpClient` calls `GetAsync` and then `ReadAsByteArrayAsync`
on both `FetchDidLogAsync` and `FetchWitnessFileAsync`. No size limit,
no streaming, no `ResponseHeadersRead`. A hostile did:webvh host can
push hundreds of MB into the resolver's memory.

### Plan

1. New `WebVhHttpClientOptions` (public) with:
   - `MaxDidLogBytes` (default `5 * 1024 * 1024` — 5 MB)
   - `MaxWitnessFileBytes` (default `1 * 1024 * 1024` — 1 MB)
2. `DefaultWebVhHttpClient` accepts an optional `WebVhHttpClientOptions`.
3. Replace each fetch with:
   - `GetAsync(url, HttpCompletionOption.ResponseHeadersRead, ct)`
   - If `Content-Length > limit` → return null
   - Stream into a bounded buffer via `ReadAtMostAsync(stream, limit, ct)`.
     If the body exceeds `limit + 1` bytes during read, return null.
4. Document the defaults in XML doc comments.

### Tests (new file: `tests/NetDid.Method.WebVh.Tests/DefaultWebVhHttpClientTests.cs`)

Use a small in-test `HttpMessageHandler` subclass to control responses:

1. Normal small `did.jsonl` → returns bytes.
2. `Content-Length` header above `MaxDidLogBytes` → returns null.
3. No `Content-Length`, body above `MaxDidLogBytes` → returns null (stream aborts).
4. Same three cases for witness file at its lower limit.
5. Non-success status → null.
6. `HttpRequestException` → null (existing behavior preserved).

## Issue #48 — conformance scope clarification

### Summary

The generated `w3c-conformance-report.md` reports per-method totals from
the local DID Core harness, but it can be misread as full method
conformance certification. The audit findings (#50, #49, #52) were not
caught by it because they live in per-method test projects.

PRs #54, #56, #55 already added the missing negative tests
(`LogChainValidatorAuthorizationTests`, `Issue49_*` in
`DidUrlMapperTests`, `Issue52_*` in `DidPeerMethodTests`). What's
remaining is the report itself: make scope explicit.

### Plan

Edit `tests/NetDid.Tests.W3CConformance/Infrastructure/ConformanceReport.cs::GenerateMarkdownReport`
to add a "Scope and limitations" section after the summary table:

- Clarify this is **local DID Core 1.0 coverage**, not official method conformance certification.
- Point at the per-method negative test classes that cover the recent audit findings.
- Note that hostile input categories live in those method-specific suites.

No new conformance suite tests — the negative coverage already exists
in the right places per #48's acceptance criteria.
