## Scope (one line)

Expose a public `IVerificationRelationshipResolver` that answers: given a controller DID, a VM DID URL, and a verification relationship, does the **controller's** DID document authorize that VM for that relationship?

## Acceptance test cases (mirror of plan)

1. Cross-DID reference (Break A) → Authorized
2. Relationship discrimination (Break B) → NotAuthorized for the wrong relationship
3. Embedded entry → Authorized via `EmbeddedMethod.Id`
4. Normalization — fragment-only entry `#k1` → Authorized
5. Normalization — bare id `k1` → Authorized
6. Empty/missing relationship → NotAuthorized
7. Resolution failure (`notFound`, `invalidDid`, `methodNotSupported`) → ControllerNotResolvable with ResolutionError populated
8. did:key Ed25519 → 4 relationships Authorized; X25519 → only KeyAgreement
9. Wrong controller — absolute `did:example:B#k1` under doc A authorizes; `#k1` under A does NOT match `did:example:B#k1`
10. Argument guards on null/whitespace inputs

**Limitations documented (intentional):**

- Query/path components inside `Reference` strings are NOT stripped — exact normalized-string match (tested explicitly).
- Resolver does not walk the resolved document's own `controller` list. Multi-controller callers iterate per controller and OR-combine.
- No internal cache; relies on `CachingDidResolver` if wired by the consumer.

**Deferred (per user direction):** `NetDidVersion` stays at 1.3.1; bump happens at release prep, not feature merge.
