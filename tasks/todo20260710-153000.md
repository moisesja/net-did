# Issue #80 — did:webvh HTTP client has no configurable timeout (100s default)

Issue: https://github.com/moisesja/net-did/issues/80
Branch: claude/net-did-issue-80-5bf2b5

## Assessment: VALID

- `DefaultWebVhHttpClient.cs:20` — bare `new HttpClient()` fallback keeps the framework
  default 100s `Timeout`.
- `NetDidBuilder.cs:50` — `AddHttpClient<DefaultWebVhHttpClient>()` registers the typed
  client with no timeout configuration (also 100s default).
- `WebVhHttpClientOptions.cs` — bounds bytes only; no time knob.
- Extra finding beyond the issue text: with `HttpCompletionOption.ResponseHeadersRead`,
  `HttpClient.Timeout` only bounds time-to-headers. The subsequent
  `ReadAsStreamAsync`/`ReadAsync` body reads are NOT covered by `HttpClient.Timeout`,
  so a host that drips the body slowly (staying under `MaxDidLogBytes`) can hold the
  fetch open indefinitely — worse than the 100s the issue describes.

## Design

Enforce a **total per-fetch timeout** (headers + body) inside `FetchBoundedAsync` via a
linked `CancellationTokenSource` + `CancelAfter(options.Timeout)`. This single
enforcement point:

- covers BOTH construction paths (bare fallback and DI typed client) without touching
  `NetDidBuilder` — the options singleton already flows into the typed client's ctor
  via `ActivatorUtilities`;
- covers the slow-drip-body gap that `HttpClient.Timeout` cannot;
- keeps `HttpClient.Timeout` (100s default) as an untouched outer backstop.

Timeout expiry (caller token NOT cancelled) normalizes to a failed fetch (`null` →
`notFound`), exactly matching the #81 contract already documented in
`DidWebVhMethod.cs:318-320`. Genuine caller cancellation keeps propagating.

Default: 30 seconds (issue suggests 10–30s). `Timeout.InfiniteTimeSpan` disables.

## Review

Implemented as designed: `WebVhHttpClientOptions.Timeout` (default 30s,
`Timeout.InfiniteTimeSpan` disables) enforced per-fetch in
`DefaultWebVhHttpClient.FetchBoundedAsync` via linked CTS + `CancelAfter`, with
`catch (OperationCanceledException) when (!ct.IsCancellationRequested) → null`.
No `NetDidBuilder` change needed — the existing DI test pattern proved the options
singleton reaches the typed client, and a new DI test
(`AddDidWebVh_FlowsCustomTimeoutIntoClient`) pins that for `Timeout` specifically.

Adversarial subagent review verdict: fix fundamentally sound; no critical/high/medium
findings. Two low findings were fixed: (1) negative `Timeout` threw
`ArgumentOutOfRangeException` from `CancelAfter` *outside* the try — escaping the
"returns null on failure" contract and being silently swallowed into all-`notFound`
by `DidWebVhMethod`'s broad catch; (2) `TimeSpan.Zero` silently cancelled every fetch.
Both closed by validating `Timeout` in the options init accessor (positive or
`InfiniteTimeSpan`, else `ArgumentOutOfRangeException` at construction), with tests.
Info-level notes (pre-existing, out of scope, left alone): cancellation surfacing as
`HttpRequestException` from a custom handler would normalize to null; the singleton
`IWebVhHttpClient` captures one typed client for app lifetime (stale-handler
anti-pattern); DNS `getaddrinfo` blocking is a general .NET limitation — the fetch
still returns within the timeout. InfiniteTimeSpan opt-out doc wording strengthened
to spell out that it removes ALL body-read bounds.

Tests added (8): withheld-headers timeout (log + witness), stalled-body timeout,
caller-cancellation propagation, default pinned at 30s, zero/negative rejected,
infinite accepted, DI timeout flow. Full suite: 798 passed / 0 failed.
