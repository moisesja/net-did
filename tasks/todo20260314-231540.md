## Working Notes

- `dotnet test netdid.sln --no-restore` passed: 573 tests, 0 failures.
- Initial confirmed findings are concentrated in:
  - `did:key` / `did:peer:0` elliptic-curve public key encoding
  - `did:peer:2` identifier and service encoding semantics
  - `did:peer:4` long-form encoding algorithm and API shape
  - shared DID resolution and DID URL dereferencing behavior

## Findings

- High: `did:peer:4` long-form encoding and contextualization diverge from the current peer-DID spec. The implementation hashes raw JSON-LD bytes, base64url-encodes raw JSON instead of multibase-encoding a multicodec-wrapped document, requires placeholder root IDs, and omits the short form from `alsoKnownAs`.
  - Issue: `#25`
  - File: `src/NetDid.Method.Peer/Numalgo4Handler.cs`

- High: `did:key` and `did:peer:0` use uncompressed EC point encodings for `P-256`, `P-384`, and `secp256k1`, and resolution does not validate key payload length/shape.
  - Issue: `#27`
  - Files: `src/NetDid.Core/Crypto/DefaultKeyGenerator.cs`, `src/NetDid.Method.Key/DidKeyMethod.cs`, `src/NetDid.Method.Peer/Numalgo0Handler.cs`

- High: `did:peer:2` uses non-spec verification method and service identifier materialization and silently drops explicit service IDs.
  - Issue: `#28`
  - Files: `src/NetDid.Method.Peer/Numalgo2Handler.cs`, `src/NetDid.Method.Peer/Numalgo2ServiceEncoder.cs`

- Medium: `DefaultDidUrlDereferencer` only implements a narrow subset of the DID Resolution service dereferencing algorithm.
  - Issue: `#29`
  - Files: `src/NetDid.Core/Resolution/DefaultDidUrlDereferencer.cs`, `src/NetDid.Core/Model/DidUrlDereferencingResult.cs`, `src/NetDid.Core/Model/DidUrlDereferencingOptions.cs`

- Medium: `DidManager` and `CompositeDidResolver` misclassify invalid DID syntax as `methodNotSupported`.
  - Issue: `#26`
  - Files: `src/NetDid.Core/DidManager.cs`, `src/NetDid.Core/Resolution/CompositeDidResolver.cs`

## Review Summary

- Current automated coverage does not detect these issues; `dotnet test netdid.sln --no-restore` still passes with 573 passing tests.
- The highest-risk problems are interoperability failures where the library produces method identifiers or long-form artifacts that are internally self-consistent but diverge from the active specifications.
- The shared dereferencing/resolution stack also has standards gaps that can mislead callers about the type or validity of the resource they received.
