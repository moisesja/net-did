## Findings

- `#27` Partially addressed.
  - Fixed:
    - generated `P-256`, `P-384`, and `secp256k1` keys are now compressed in `DefaultKeyGenerator`
    - `did:key` and `did:peer:0` resolution now reject invalid key lengths
    - JWK conversion now understands compressed SEC1 points
  - Still missing:
    - create paths still trust caller-supplied `ExistingKey.PublicKey` bytes without normalizing EC keys to the spec form
    - resolution validates length only, not point validity/shape, so invalid compressed EC points can still resolve into DID Documents

- `#28` Partially addressed.
  - Fixed:
    - verification method numbering is now 1-based
    - explicit fragment service IDs are preserved
    - auto-generated service IDs now follow `#service`, `#service-1`, ...
  - Still missing:
    - verification method and service IDs are still materialized as absolute DID URLs (`{did}#key-1`, `{did}#service`) instead of the spec-defined relative IDs (`#key-1`, `#service`)
    - new tests now assert the absolute-ID behavior, which locks in the remaining divergence

- `#25` Partially addressed.
  - Fixed:
    - long-form payload is now multibase/base58btc with a JSON multicodec prefix
    - short-form hash is now derived from the long-form encoded string
    - `alsoKnownAs` now includes the short-form DID
  - Still missing:
    - numalgo 4 still serializes the input document as JSON-LD directly from `DidDocument`, rather than encoding a properly contextualized pre-resolution document
    - the public API still requires a root `DidDocument.Id`, so callers still need the placeholder-ID workflow that the issue called out
    - tests still model numalgo 4 input using `did:peer:placeholder`, which suggests the placeholder-based flow remains part of the design rather than being eliminated
