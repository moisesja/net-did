# Issue #97 ŌĆö Dispose generated KeyPair in DidKeyMethod/Numalgo0Handler (ND-E8 follow-up)

## Context

NetCrypto 1.2.0 (pinned in `Directory.Packages.props:9`) made `KeyPair` `IDisposable`:
`Dispose()` deterministically zeroizes the pinned private-key copy (idempotent; key-material
access afterwards throws `ObjectDisposedException`), and `PublicKey` returns a defensive copy
on every get (verified against the package XML docs).

Two fresh-key create paths generate a `KeyPair`, read only `PublicKey`, and orphan the private
half to the GC unwiped ŌĆö defeating the deterministic-wipe guarantee:

1. `src/NetDid.Method.Key/DidKeyMethod.cs:57-58` (`CreateCoreAsync`, fresh-key branch)
2. `src/NetDid.Method.Peer/Numalgo0Handler.cs:36-37` (`Create`, fresh-key branch)

**Severity: Medium** ŌĆö security hygiene (private key material lingers in process memory until
GC); no remote exploitability, no functional breakage.

**Scope check (verified):** these are the only two `.Generate(` call sites under `src/`
(grep). Out of scope:
- `ExistingKey` branches in both files ŌĆö `ExistingKey` is `ISigner` (caller-owned key
  custody); no `KeyPair` is created or owned there, nothing to dispose.
- `DeriveX25519PublicKeyFromEd25519` at `DidKeyMethod.cs:155` ŌĆö returns a
  `PublicKeyReference` (public-only, no private material, not disposable).
- All other key custody routes through `IKeyStore`/`ISigner`, where NetCrypto 1.2.0's
  store-side disposal semantics apply.

**Plan-approval note:** fix is two one-line `using var` insertions, exactly as prescribed by
the maintainer-authored issue ŌĆö treated as a simple, obvious fix (CLAUDE.md ┬¦5), executed
autonomously on the pre-created issue branch `claude/github-issue-97-b20117`.
