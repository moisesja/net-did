## dataproofs-dotnet (DataProofsDotnet)

### [BLOCKER] DP-1: DataProofsDotnet.Core is not published to NuGet
- **Why net-did needs it:** net-did's entire Data Integrity / did:webvh phase references `DataProofsDotnet.Core` (which transitively pins NetCrypto 1.0.0-preview.1 and NetCid 1.6.0). The package is absent from the local NuGet cache and there is no `.nupkg` in the repo; the version is `0.1.0-preview.1` and the project is "being finalized". net-did's CI cannot `restore` against it, so the migration's last (and riskiest) phase cannot begin.
- **Proposed API/behavior:** Publish `DataProofsDotnet.Core` (at minimum) to a feed net-did can restore from, with a documented minimum version and its transitive NetCrypto/NetCid floors. Freeze the `EddsaJcs2022Cryptosuite` / `DataIntegrityProof` / `PublicKeyMaterial` / `ProofVerificationResult` public surface in `PublicAPI.Shipped.txt` so net-did can pin a stable contract.
- **Severity:** blocker

### [BLOCKER] DP-Blocker: eddsa-jcs-2022 wire-format mismatch vs net-did's current did:webvh signing
- **Why net-did needs it:** net-did's current `DataIntegrityProofEngine` signs `JCS(document-only)` bytes (verified `DataIntegrityProofEngine.cs:43-44`), whereas DataProofs' conformant `EddsaJcs2022Cryptosuite` signs `hashData = SHA-256(canonicalProofConfig) ‖ SHA-256(canonicalDocument)` (verified `EddsaJcs2022Cryptosuite.cs`/`JcsCryptosuite.cs`). Existing did:webvh log entries and witness proofs produced by net-did will FAIL verification under DataProofs, and vice versa — a hard interop break. did:webvh LogEntry has no `@context`, so the proof config is hashed without `@context`, a signing input net-did never produced.
- **Proposed API/behavior:** This is primarily a net-did/spec decision, not a code gap: adopt the conformant hash-concat algorithm for did:webvh going forward (version-bump the log, regenerate fixtures). File upstream ONLY if did:webvh genuinely requires a document-only JCS sign mode — in which case request an opt-in document-only cryptosuite variant in DataProofsDotnet. Preferred resolution: migrate to the conformant suite.
- **Severity:** blocker

### DP-2: No DID-method-aware verificationMethod parser / authorized-key extractor (keep in net-did)
- **Why net-did needs it:** `LogChainValidator` relies on net-did's `ExtractDidKeyMultibase` for security-critical exact-ordinal matching of the proof signer's multibase against the entry's authorized `updateKeys`, enforcing the `did:key`==fragment anti-confusion rule (defends `did:key:<attacker>#<authorized>` spoofing). DataProofs exposes only the opaque-URL `IVerificationMethodResolver` and `PublicKeyMaterial.FromMultikey`; neither parses a `did:key` URL into its multibase id nor enforces DID==fragment ordinal equality.
- **Proposed API/behavior:** No upstream change — the dependency direction (net-did -> dataproofs) forbids a DID-aware parser upstream. net-did retains/relocates the parser into its `did:key` method and feeds the extracted multibase into `PublicKeyMaterial.FromMultikey`. Filed for documentation: confirm DataProofs intends consumers to own DID-URL parsing and only accept resolved `PublicKeyMaterial`.
- **Severity:** major (resolved net-did-side; no upstream code change)

### DP-3: did:webvh allow-list authorization is not the resolver's controller model
- **Why net-did needs it:** did:webvh authorizes by an explicit `updateKeys` allow-list in log parameters, NOT by controller-document proof-purpose relationships. The resolver-driven `DataIntegrityProofPipeline.VerifyAsync` performs controller authorization that does not map to did:webvh, and it returns an aggregate `DocumentVerificationResult` (all-proofs-must-verify), which is the wrong semantics for did:webvh's "accept first valid authorized proof" loop.
- **Proposed API/behavior:** No upstream change — use the suite-level raw-key `EddsaJcs2022Cryptosuite.VerifyProof(...).Verified` and keep net-did's allow-list check. Filed for documentation: confirm the suite-level raw-key verify is a supported, stable entry point distinct from the pipeline, so net-did can rely on it.
- **Severity:** minor (resolved net-did-side)

## crypto-dotnet (NetCrypto)

### CD-1: NetCrypto is preview-only (1.0.0-preview.1); no stable channel
- **Why net-did needs it:** Adopting NetCrypto forces net-did to take a prerelease dependency (requires explicit `--prerelease`/nuget.config opt-in) for its entire crypto substrate. The `ICryptoProvider`/`DefaultCryptoProvider`/`IBbsCryptoProvider`/`KeyType`/signer/keystore/JWK surface is already frozen in `PublicAPI.Shipped.txt`, so the contract is stable; only the release channel is a supply-chain/stability concern for net-did pinning.
- **Proposed API/behavior:** Stabilize NetCrypto to a non-preview `1.0.0` (or commit to a GA timeline) so net-did is not forced to ship against a prerelease. No API change required.
- **Severity:** minor

### CD-2: No public BBS keygen helper (net-did BBS test cannot be ported 1:1)
- **Why net-did needs it:** net-did's `DefaultBbsCryptoProviderTests` mints BBS keypairs by calling the internal `ZkryptiumNative.bbs_keygen` directly. After migration that FFI is internal to NetCrypto (`InternalsVisibleTo` only `NetCrypto.Tests`), so `NetDid.Core.Tests` has no public raw-FFI keygen path. The public surface offers only `DefaultBbsCryptoProvider.Sign/Verify/DeriveProof/VerifyProof` and `DefaultKeyGenerator.Generate(KeyType)` (returns a `KeyPair`, not the raw 32/96 sk/pk pair the test uses).
- **Proposed API/behavior:** No upstream change strictly required — net-did deletes its BBS test (BBS is now NetCrypto's responsibility and is covered by `NetCrypto.Tests`). Optionally, NetCrypto could expose a public test-only BBS keygen helper; not recommended given the frozen-surface posture. Document that BBS keygen for consumers is `DefaultKeyGenerator.Generate(KeyType.Bls12381G2)`.
- **Severity:** minor

### CD-3: EcPointValidatorTests reach into internal DefaultCryptoProvider decompress methods
- **Why net-did needs it:** `tests/NetDid.Core.Tests/Crypto/EcPointValidatorTests.cs` calls `DefaultCryptoProvider.DecompressEcPoint` / `DecompressSecp256k1Point` directly (`:207,:225`). These are `internal` in both repos; in NetCrypto they are visible only to `NetCrypto.Tests`. Post-migration `NetDid.Core.Tests` cannot reach them, so those specific test cases cannot be rewired with a using-swap.
- **Proposed API/behavior:** No upstream change preferred — net-did rewrites those cases against the public `EcPointValidator.EnsureOnCurve` API (which exercises the same validation) or deletes them (duplicated in `NetCrypto.Tests`). If a public decompression primitive is genuinely useful, file a request for `NetCrypto` to expose a public `DecompressEcPoint`; not recommended.
- **Severity:** minor

### CD-4 (informational): IBbsCryptoProvider surface drift + BBS-absent exception change
- **Why net-did needs it:** `NetCrypto.IBbsCryptoProvider` adds `Ciphersuite` + `IsAvailable` members net-did's interface lacked, and the BBS-absent path throws `NetCrypto.BbsUnavailableException` (a `CryptographicException`) instead of net-did's `PlatformNotSupportedException`. net-did has zero in-src BBS consumers, so this is consume-only, but any future net-did adapter implementing the interface must supply the extra members and probe `IsAvailable`.
- **Proposed API/behavior:** No upstream change — adopt NetCrypto's richer interface as-is. Document in net-did migration notes that BBS availability is probed via `IsAvailable` (never throws) and failures surface as `BbsUnavailableException`.
- **Severity:** minor

## net-cid (NetCid)

### NC-1 (informational): JcsCanonicalizer has no string-in/string-out overload
- **Why net-did needs it:** net-did's only production JCS consumer (`ScidGenerator`) passes/handles JSON strings, while `NetCid.JcsCanonicalizer.Canonicalize` accepts only `JsonElement`/`JsonNode` and returns `byte[]`. Each call site must `JsonDocument.Parse(...)` first (a one-liner; placeholders inside string values survive parsing). NetCid 1.6.0 (verified in cache) is also a prerequisite bump from net-did's current 1.5.0 for the hardened number serialization + depth/duplicate/surrogate/output guards.
- **Proposed API/behavior:** Recommend declining a string convenience overload (keep NetCid lean); net-did absorbs the parse at its 2 call sites. The required action is purely net-did's: bump NetCid 1.5.0 -> 1.6.0. No upstream code change.
- **Severity:** minor
