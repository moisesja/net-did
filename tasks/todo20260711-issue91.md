# Issue #91 — Expose update-key rotation evidence on `DidUpdateResult`; ship as v2.2.0

## Context

Issue [#91](https://github.com/moisesja/net-did/issues/91): `DidUpdateResult.AuthorizationChange` (added in 2.1.0 for #82) reports `Changed` when **any** of did:webvh's `updateKeys`, `nextKeyHashes`, `prerotation`, or witness config changes. A method-agnostic consumer (net-wallet-sdk `RotateAsync`, PR #15) therefore cannot (1) prove the active update key actually rotated vs. a policy-only change, nor (2) verify its new key is in the newly authorized set — the effective `updateKeys` live in webvh log parameters the abstract layer never sees.

**Fix (additive, backward-compatible, per the issue's proposed contract):** add two properties to `DidUpdateResult`:
- `AuthorizationChangeStatus UpdateKeyChange` — specifically whether the effective `updateKeys` set changed (reuses the existing fail-closed tri-state; default `Unknown`).
- `IReadOnlyList<string>? EffectiveUpdateKeys` — the effective `updateKeys` of the new latest log entry, i.e. the multibase keys authorized to sign the **next** log entry (update or deactivation). `null` = method didn't report; empty = no keys authorized (DID frozen).

`AuthorizationChange` keeps its current coarse meaning — no behavioral change to any shipped surface.

Verified semantics (LogChainValidator.cs:63-72, 142-147; DidWebVhMethod.cs:392-394): entry N's proof is authorized against updateKeys effective as of entry N-1, so the new entry's effective set is exactly "keys authorized to sign the next entry". At the result-construction point `newEffectiveParams.UpdateKeys` is never null (signer authorization at DidWebVhMethod.cs:393 guarantees a non-null pre-merge set; `MergeWith` carries non-null forward).

Ship as **v2.2.0** end-to-end: feature PR → merge on green CI → release PR → merge → tag `v2.2.0` → NuGet publish via `.github/workflows/publish.yml` (triggers on `v*` tag push).

### 2. Core model — `src/NetDid.Core/Model/DidUpdateResult.cs`
- Add `public AuthorizationChangeStatus UpdateKeyChange { get; init; }` with XML doc: whether the effective `updateKeys` set changed; fail-closed `Unknown` default; `Changed` does **not** imply the old key lost authority (additive updates exist) — a rotation postcondition is `UpdateKeyChange == Changed` **and** new key ∈ `EffectiveUpdateKeys` **and** retired key ∉ `EffectiveUpdateKeys`.
- Add `public IReadOnlyList<string>? EffectiveUpdateKeys { get; init; }` with XML doc: keys authorized to sign the next log entry (update or deactivation), **not** the keys that authorized this update; `null` = not reported (fail closed), empty = no keys authorized (DID frozen).

### 4. Tests
`tests/NetDid.Core.Tests/AuthorizationChangeStatusTests.cs` (mirror existing fail-closed facts):
- Default `DidUpdateResult` → `UpdateKeyChange == Unknown`, `EffectiveUpdateKeys == null`.
- Non-reporting method via `DidMethodBase.UpdateAsync` → same.

`tests/NetDid.Method.WebVh.Tests/DidWebVhMethodTests.cs` — new `Issue91_*` facts alongside the `Issue82_*` block (~line 1996):
1. **Rotation (replace)**: `UpdateKeys=[k2]` → `UpdateKeyChange=Changed`, `AuthorizationChange=Changed`, `EffectiveUpdateKeys == [k2]`, old key absent.
2. **Additive**: `UpdateKeys=[k1,k2]` → `UpdateKeyChange=Changed`, `EffectiveUpdateKeys` still contains `k1` (pins the "Changed ≠ revocation" doc caveat).
3. **Witness-only change** → `AuthorizationChange=Changed`, `UpdateKeyChange=Unchanged`, `EffectiveUpdateKeys` == carried-forward set. *(the headline #91 discriminator)*
4. **Prerotation/nextKeyHashes-only** → same as 3.
5. **Doc-only edit** and **NewDocument=null preserve path** → both statuses `Unchanged`, `EffectiveUpdateKeys` non-null == genesis key.
6. **Same keys re-supplied** → `UpdateKeyChange=Unchanged`.
7. **Order/duplicate insensitivity**: prior `[k1,k2]`, supply `[k2,k1]` → `Unchanged`.
8. **Pre-rotation forced rotation**: commit hash of k2, update signed by k1 with `UpdateKeys=[k2]` → `Changed`, `EffectiveUpdateKeys=[k2]`.
9. **Pre-rotation recommit-to-same-key**: commitment for the *current* key, re-supply it → passes `ValidateKeyRotation` yet `UpdateKeyChange=Unchanged` (proves the wallet can detect a phantom rotation).
10. **Empty set**: `UpdateKeys=[]` → `Changed`, `EffectiveUpdateKeys` empty-non-null (pins freeze behavior).
11. **Defensive copy**: mutate the caller's `List<string>` after return → reported evidence unchanged.
12. **Writer/reader parity**: `ValidateChain` over the returned log yields an effective `UpdateKeys` set-equal to `EffectiveUpdateKeys`.
13. Invariant across scenarios: `UpdateKeyChange=Changed` ⇒ `AuthorizationChange=Changed`.

## Review round 1 (PR #92) — 2026-07-11

Maintainer review found 2 High, 1 Medium, 1 Low. All addressed:

1. **High — evidence contract false under pre-rotation** (v1.0 authorizes a pre-rotation
   entry with the current entry's OWN updateKeys; NetDid's model deviates — filed #93).
   → Key evidence now fails closed (`UpdateKeyChange = Unknown`, `EffectiveUpdateKeys = null`)
   whenever pre-rotation is in play before or after the update. Pre-rotation tests rewritten
   to pin the withheld evidence.
2. **High — dynamic IReadOnlyList TOCTOU** could desync signed artifact from reported
   evidence. → All caller collections (UpdateKeys, NextKeyHashes, witness list) snapshotted
   once in BuildUpdateParameters/SnapshotWitnessConfig; every stage reads the private copy.
   Regression test `Issue91_Update_DynamicKeyList_CannotDesyncEvidenceFromArtifact` added and
   proven to fail against the unsnapshotted code.
3. **Medium — membership postcondition admits supersets.** → Docs (XML, README, PRD,
   CHANGELOG) now require set-equality with the intended post-rotation set for exclusive
   rotation.
4. **Low — stale empty-set warning.** → v1.0 explicitly permits `updateKeys: []` (frozen
   DID); docs/PR body corrected.

942 tests pass, 0-warning Release build.
