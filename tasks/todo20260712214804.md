# Issue #98 — Report validated rotation evidence during continuous pre-rotation

## Context

Issue [#98](https://github.com/moisesja/net-did/issues/98) is confirmed High priority on clean
`main` at `44b0844`.

`DidWebVhMethod.UpdateCoreAsync` already snapshots the caller's update collections, validates every
explicit pre-rotation `updateKeys` member against the prior commitments, and restricts the proof
signer to that validated set (`src/NetDid.Method.WebVh/DidWebVhMethod.cs:440-485`). It also computes
the effective before/after update-key set comparison unconditionally (`DidWebVhMethod.cs:499-503`).
The information loss occurs later: when the resulting state retains non-empty `nextKeyHashes`, the
known comparison is replaced with `UpdateKeyChange = Unknown`, `EffectiveUpdateKeys` remains null,
and the validated `authorizedKeys` set is omitted from the result (`DidWebVhMethod.cs:505-576`).

The did:webvh v1.0 normative rules distinguish the same two concepts:

- without pre-rotation, the prior effective `updateKeys` authorize the current entry;
- with pre-rotation, the current entry's explicit `updateKeys` authorize it, every member must match
  a prior commitment, and one of those active keys must verify the proof;
- non-empty current `nextKeyHashes` constrain a future entry but do not reveal which committed
  preimages will actually be used.

Sources: the official v1.0
[`updateKeys` / `nextKeyHashes` parameters](https://identity.foundation/didwebvh/v1.0/#didwebvh-did-method-parameters),
[`Authorized Keys`](https://identity.foundation/didwebvh/v1.0/#authorized-keys), and
[`Pre-Rotation`](https://identity.foundation/didwebvh/v1.0/#pre-rotation-key-hash-generation-and-verification)
sections. Therefore the minimal conformant fix is result/API-only: report the complete set eligible
to authorize the entry just appended, compute `UpdateKeyChange` from the fully known effective sets,
and continue withholding only the unknowable next-entry set.

Explicitly out of scope:

- `PreRotationManager`, `LogChainValidator`, proof creation/verification, entry hashing, SCID,
  serialization, and log wire format: their validation/state-machine behavior is already correct.
- Resolution, create, DID routing/DI, `did:key`, and `did:peer`: no behavior change is required;
  non-reporting drivers retain fail-closed defaults.
- `DidDeactivateResult` / `DeactivateCoreAsync`: issue #98's required property and acceptance criteria
  concern `DidUpdateResult`. Adding a separate deactivation evidence API would expand the public
  design without a specified consumer contract.
- Witness-threshold/publication evidence: the new property attests only to the complete update-key
  set eligible to authorize the appended entry, not that every listed key signed or that all witness
  requirements have been satisfied by the unpublished artifact.

## Review

### Implementation

- Added fail-closed `DidUpdateResult.RevealedUpdateKeys` for the complete set eligible to authorize
  the entry just appended. did:webvh reports a defensive read-only copy of the same validated
  `authorizedKeys` snapshot used to restrict the signer.
- `UpdateKeyChange` now always reports the known effective `updateKeys` set comparison. Fresh
  commitments still keep only the forward-looking `EffectiveUpdateKeys` at `null`.
- No log serialization, wire format, SCID, hash, proof, resolver, or deactivation behavior changed.
- Updated public API docs, README guidance, PRD source of truth, `[Unreleased]` changelog, fail-closed
  Core coverage, existing #91/#93 expectations, and new `Issue98_*` regression/security tests.

### Acceptance criteria and behavior diff vs. `main`

- Continuous pre-rotation (reveal committed key + fresh recommit): `UpdateKeyChange == Changed`,
  `RevealedUpdateKeys` set-equals the current explicit commitment-validated set, and
  `EffectiveUpdateKeys == null`.
- Ordinary rotation preserves its existing `Changed` plus concrete next-entry
  `EffectiveUpdateKeys`; the additive `RevealedUpdateKeys` correctly reports the prior set that
  authorized the appended entry.
- Commitment-only activation and same-key recommit report `Unchanged`, preventing phantom rotations.
- A pre-rotation exit preserves #91/#93 behavior: the entry is authorized by its current revealed
  committed set and its concrete effective set is available for the successor.
- An ordinary entry that simultaneously changes `updateKeys` and activates pre-rotation is pinned as
  `Changed / Effective null / prior Revealed`; docs explicitly prohibit coalescing current-entry and
  next-entry evidence into a generic post-change set.
- Non-reporting drivers retain `Unknown / null / null` fail-closed defaults.
