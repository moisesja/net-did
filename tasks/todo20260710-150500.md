# Issue #81 — did:webvh resolution swallows caller cancellation

## Assessment

**Valid.** `DidWebVhMethod.ResolveCoreAsync` (src/NetDid.Method.WebVh/DidWebVhMethod.cs:329)
ends with `catch (Exception ex) → DidResolutionResult.NotFound(did)`. The try block awaits
`_httpClient.FetchDidLogAsync(logUrl, ct)` / `FetchWitnessFileAsync(witnessUrl, ct)`;
`DefaultWebVhHttpClient` passes `ct` to `HttpClient.GetAsync` / `ReadAsStreamAsync` /
`Stream.ReadAsync` and only catches `HttpRequestException`, so a caller-cancelled token
surfaces as `OperationCanceledException`/`TaskCanceledException` in `ResolveCoreAsync` and
the broad catch converts it into a `notFound` resolution result. The caller cannot
distinguish "DID doesn't exist" from "I cancelled" — this breaks the cooperative
cancellation contract. Nothing upstream (`DidMethodBase.ResolveAsync`, `DidManager`,
`CompositeDidResolver`, `CachingDidResolver`) re-throws or masks, so the swallow is real
end-to-end.

Out of scope (per the issue, confirmed): `DidKeyMethod.cs:107`, `DidPeerMethod.cs:87`, and
`WebVhProofVerifier.cs:62` are CPU-bound and never observe `ct`, so no
`OperationCanceledException` can reach those catches — adding the guard there would be dead
code. `HttpClient.Timeout`-driven `TaskCanceledException` (caller token NOT cancelled) must
keep normalizing to a resolution failure — that behavior belongs to open issue #80.

## Plan

- [x] Add `catch (OperationCanceledException) when (ct.IsCancellationRequested) { throw; }`
      ahead of the existing catches in `ResolveCoreAsync`
- [x] Test: cancelled caller token → `ResolveAsync` throws `OperationCanceledException`
- [x] Test: `TaskCanceledException` from the HTTP layer with a non-cancelled caller token
      (HttpClient.Timeout shape) → still `notFound` (regression guard for the `when` filter)
- [x] Build + run full test suite
- [x] Update CHANGELOG.md `[Unreleased]` → Fixed
- [x] Review section below

## Review

- Fix is a single catch clause in `DidWebVhMethod.ResolveCoreAsync` (before the
  `LogChainValidationException` and broad catches). No API/behavior change other than
  cancellation now propagating.
- Two new tests added in `DidWebVhMethodTests` with a `ThrowingWebVhHttpClient` stub:
  - `Issue81_Resolve_CallerCancellation_PropagatesOperationCanceledException`
  - `Issue81_Resolve_HttpTimeoutWithoutCallerCancellation_ReturnsNotFound`
- Behavior diffed against old code: with the fix stashed, the cancellation test FAILS
  (old code returned `notFound` for a cancelled token) and the timeout test still passes;
  with the fix restored, both pass. Full suite: 789/789 green.
- `w3c-conformance-report.md` timestamp churn from the conformance test run was reverted.
- Deliberately NOT touched: `DidKeyMethod`/`DidPeerMethod`/`WebVhProofVerifier` broad
  catches — CPU-bound, never observe `ct`, guard there would be dead code (minimal-impact
  principle; issue itself calls this optional).
